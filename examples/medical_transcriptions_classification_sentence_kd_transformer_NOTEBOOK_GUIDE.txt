================================================================================
 NOTEBOOK GUIDE
 examples/medical_transcriptions_classification_sentence_kd_transformer.ipynb
================================================================================

Plain-English walk-through of every cell in the companion notebook for the
SentenceKDTransformer contribution. The notebook now has FULL PARITY with
the companion .py script's ablation suite -- every experiment in the PR
description is runnable interactively here.

Paper:    Kim et al., "Integrating ChatGPT into Secure Hospital Networks: A
          Case Study on Improving Radiology Report Analysis", CHIL 2024.
Model:    pyhealth/models/sentence_kd_transformer.py
Script:   examples/medical_transcriptions_classification_sentence_kd_transformer.py
Tests:    tests/core/test_sentence_kd_transformer.py
Docs:     docs/api/models/pyhealth.models.SentenceKDTransformer.rst


--------------------------------------------------------------------------------
 HIGH-LEVEL GOAL
--------------------------------------------------------------------------------
Give a reviewer a single Colab-runnable artifact that demonstrates the
entire contribution: model instantiation, all four ablations, the saved
structured results, and the paper's document-level aggregation helper.

Four ablations, matching the companion .py:

  Ablation 1 (novel) - lambda sweep (contrastive-loss weight grid)
  Ablation 2 (novel) - tau sweep (contrastive temperature grid)
  Ablation 3 (rubric) - lr x dropout x batch size grid (3 x 2 x 2 = 12 cells)
  Ablation 4 (novel) - backbone comparison paired with lambda = 1

All data is synthetic (no DUA, no MIMIC-CXR credentials). The backbone
is StanfordAIMI/RadBERT (~110M parameters, ~440 MB first-run download),
with bert-base-uncased and Bio_ClinicalBERT additionally downloaded by
Ablation 4.

A GPU is required. On Colab:
    Runtime -> Change runtime type -> T4 GPU

Expected end-to-end runtime on a free Colab T4, cold HuggingFace cache:
~5-8 minutes.

Paper citation, enumerated list of the 7 things the notebook demonstrates

--------------------------------------------------------------------------------
 CELL 1 (code) -- COLAB-AWARE INSTALL + PYARROW REPAIR + REPO CLONE
--------------------------------------------------------------------------------
Environment bootstrap. Four responsibilities:

1. Detect Colab via `import google.colab`.

2. Install PyHealth if missing:
       git+https://github.com/raymondcoding15/PyHealth.git@DL4H_final

3. Self-heal pyarrow. Colab pre-loads an older pyarrow; PyHealth's deps
   can leave pyarrow.lib and pyarrow._csv compiled against different
   struct sizes ("IpcReadOptions size changed"). We probe `import
   pyarrow.csv` and rebuild it if broken. Runs every Colab session.

4. Clone the repo shallowly on Colab. Colab only downloads the .ipynb, so
   the companion .py is not on disk. We clone DL4H_final into
   /content/pyhealth_src. Locally this branch is a no-op.


--------------------------------------------------------------------------------
 CELL 2 (markdown) -- "1. Setup"
--------------------------------------------------------------------------------
Header explaining that the notebook re-uses the ablation helpers from the
.py rather than duplicating code. Single-source-of-truth rationale.


--------------------------------------------------------------------------------
 CELL 3 (code) -- IMPORTS, GPU ASSERTION, LOAD COMPANION .py
--------------------------------------------------------------------------------
Three responsibilities:

1. Standard imports (torch, pandas) and seed the RNG.

2. HARD GPU ASSERTION. RadBERT + 4 ablations on CPU would take >30 min.
   Fail fast with a clear message pointing at Runtime -> Change runtime
   type -> T4 GPU. On success, bind `DEVICE = torch.device("cuda")`.

3. Load the companion .py as a module via importlib, searching three
   candidate paths (local examples/, cwd, Colab clone location).
   `sys.modules["mt_skd"] = mt_skd` is required BEFORE `exec_module` so
   that @dataclass definitions in the .py can resolve their own module
   during class construction.


--------------------------------------------------------------------------------
 CELL 4 (markdown) -- "2. Build the dataset and verify the model"
--------------------------------------------------------------------------------
Explains the synthetic PyHealth SampleDataset source (no downloads, no
DUA).


--------------------------------------------------------------------------------
 CELL 5 (code) -- INSTANTIATE DATASET + MODEL
--------------------------------------------------------------------------------
- Binds BACKBONE = "StanfordAIMI/RadBERT" as a notebook-level constant
  shared by every downstream cell.
- `mt_skd.build_dataset(quick=True, data_root=None)` returns the
  synthetic multiclass SampleDataset.
- Instantiates SentenceKDTransformer with lam=1.0, temperature=0.07,
  max_length=128, then `.to(DEVICE)`.
- Prints output size and hidden_dim as a sanity check.


--------------------------------------------------------------------------------
 CELL 6 (markdown) -- "3. Ablation 1 -- lambda sweep [novel]"
--------------------------------------------------------------------------------
Introduces the first novel ablation. Paper toggles lambda between 0 and 1
(Table 4). We sweep {0, 0.1, 0.5, 1, 2, 5} at 1 epoch each to expose the
phase transition where the contrastive term overtakes cross-entropy.


--------------------------------------------------------------------------------
 CELL 7 (code) -- RUN LAMBDA SWEEP, DISPLAY RESULTS TABLE
--------------------------------------------------------------------------------
Calls mt_skd.run_lambda_sweep with BACKBONE, epochs=1, batch_size=16.
PyHealth's Trainer auto-moves the model to GPU. Output is silenced at
four layers (logging.disable, TQDM_DISABLE, TRANSFORMERS_VERBOSITY,
contextlib.redirect_stdout/stderr) so the visible output is only the
final DataFrame: name / accuracy / macro_f1 / loss.


--------------------------------------------------------------------------------
 CELL 8 (markdown) -- "4. Ablation 2 -- tau sweep [novel]"
--------------------------------------------------------------------------------
Motivates the second novel ablation. Paper uses tau=0.07 (Khosla default),
never varies it. We sweep {0.05, 0.1, 0.2, 0.5, 1.0} with lambda pinned
to 1 to span the "peaked" (small tau) and "uniform" (large tau) regimes
of the contrastive softmax.


--------------------------------------------------------------------------------
 CELL 9 (code) -- RUN TAU SWEEP, DISPLAY RESULTS TABLE
--------------------------------------------------------------------------------
Same pattern as cell 7 but for temperature.


--------------------------------------------------------------------------------
 CELL 10 (markdown) -- "5. Ablation 3 -- hyperparameter sweep [rubric]"  [NEW]
--------------------------------------------------------------------------------
Introduces the standard hyperparameter grid required by the DL4H rubric:
learning rate {1e-5, 3e-5, 1e-4} x dropout {0.1, 0.3} x batch size
{16, 32} = 12 cells. Runs at 1 epoch per cell; full grid completes in
~60-90 seconds on T4.


--------------------------------------------------------------------------------
 CELL 11 (code) -- RUN HYPERPARAMETER ABLATION [NEW]
--------------------------------------------------------------------------------
Calls mt_skd.run_hyperparameter_ablation and displays a DataFrame sorted
by accuracy descending so the best-performing cell sits at the top. This
is the easiest cell for a grader to spot-check.


--------------------------------------------------------------------------------
 CELL 12 (markdown) -- "6. Ablation 4 -- backbone comparison [novel]"  [NEW]
--------------------------------------------------------------------------------
Explains the novel interaction ablation between backbone choice and the
supervised contrastive signal:
    - bert-base-uncased          (generic English BERT)
    - emilyalsentzer/Bio_ClinicalBERT   (biomedical + MIMIC-III)
    - StanfordAIMI/RadBERT              (radiology, paper's primary)
Paired with lambda = 1. Paper only compares backbones in its fixed
Table 2 setup (no contrastive term). Warns that the first run downloads
~1 GB of additional weights (RadBERT already cached from cell 5).


--------------------------------------------------------------------------------
 CELL 13 (code) -- RUN BACKBONE COMPARISON [NEW]
--------------------------------------------------------------------------------
Calls mt_skd.run_backbone_comparison with quick=False so the full
3-backbone grid runs (the .py's --quick flag restricts to bert-tiny; we
explicitly override to the real clinical backbones here). Output is the
accuracy/macro-F1/loss DataFrame keyed by backbone name.


--------------------------------------------------------------------------------
 CELL 14 (markdown) -- "7. Save consolidated ablation results"  [NEW]
--------------------------------------------------------------------------------
Explains that the notebook produces the same ablations.json artifact the
.py main() writes, for end-to-end parity.


--------------------------------------------------------------------------------
 CELL 15 (code) -- WRITE ablations.json  [NEW]
--------------------------------------------------------------------------------
Serialises all four result sets (lambda_results, tau_results,
hparam_results, backbone_results) into a single JSON with keys
"lambda" / "temperature" / "hyperparam" / "backbone". Prints total cell
count and displays a small pandas Series summarising cells-per-ablation.


--------------------------------------------------------------------------------
 CELL 16 (markdown) -- "8. Paper Eq. 4 -- document-level max-pool inference"
--------------------------------------------------------------------------------
(Was section 5 in the pre-parity version of the notebook.)
Explains paper Eq. 4 (p_a(doc) = max_j p_abnormal(sentence_j)) and the
three aggregation modes: max, topk_mean, attn. Disclaims that absolute
values come from a RadBERT that hasn't been fine-tuned on ternary labels
in this demo; what matters is tensor shapes, probability normalization,
and the ordering across aggregation modes.


--------------------------------------------------------------------------------
 CELL 17 (code) -- RADIOLOGY-STYLE MIXED-FINDINGS DEMO
--------------------------------------------------------------------------------
Builds a 6-sentence ternary-labeled "radiology report" via
create_sample_dataset, instantiates a fresh RadBERT-backed
SentenceKDTransformer on it (on GPU), and tabulates the document-level
(p_abnormal, p_normal) produced by each of the three aggregation modes.


--------------------------------------------------------------------------------
 CELL 18 (code) -- PER-SENTENCE TERNARY PROBABILITY BREAKDOWN
--------------------------------------------------------------------------------
Pulls the raw per-sentence ternary probabilities out of document_predict
and displays them in a DataFrame with the original sentence text. Column
names come from rad_dataset.output_processors["label"].label_vocab so
they dynamically reflect {normal, abnormal, uncertain}. Exposes the
input to the aggregation math from cell 17 so a grader can verify by eye.


--------------------------------------------------------------------------------
 CELL 19 (markdown) -- "9. What's shown here and what's in the .py"
--------------------------------------------------------------------------------
Final summary. Updated comparison table showing that the notebook now has
parity with the .py across all four ablations and the ablations.json
artifact. The only intentional difference is epochs-per-cell (notebook
fixes epochs=1 for runtime; .py accepts --epochs). Points at the full
.py invocation for the publication-quality numbers and lists related PR
files.


--------------------------------------------------------------------------------
 TROUBLESHOOTING NOTES FOR COLAB
--------------------------------------------------------------------------------
1. BEFORE clicking Run all, set the runtime to a GPU:
     Runtime -> Change runtime type -> T4 GPU.
   If you skip this, cell 3's assertion will fail with a clear message.

2. First run on a fresh runtime:
     - Cell 1 installs pyhealth and (if needed) repairs pyarrow.
     - It prints "Packages were (re)installed. On Colab you MUST now do
       Runtime -> Restart runtime, then Run all again."
     - Runtime -> Restart runtime -> Run all.

3. Second run (after restart) on the same runtime:
     - Cell 1 is quiet, no further restart needed.
     - Cell 5 downloads RadBERT (~440 MB). HuggingFace caches it so
       cells 7/9/11/17 are instant.
     - Cell 13 downloads bert-base-uncased and Bio_ClinicalBERT
       (~440 MB each). These dominate first-run wall-clock time.
     - Total end-to-end: ~5-8 minutes on T4.

4. ablations.json is written to /content/ablations.json on Colab. Use
   the file browser (folder icon in the left sidebar) to download it
   if you want to keep a copy.

5. If you see pyarrow ABI errors again after a Colab image upgrade:
     - Manually run `!pip uninstall -y pyarrow && !pip install -q pyarrow`
       then Runtime -> Restart runtime. The install cell's self-heal will
       do this automatically on new sessions.

6. If the companion .py is not found, cell 1's clone to
   /content/pyhealth_src probably failed (rare transient GitHub 5xx).
   Rerun cell 1.
